# UX Checklist

A per-PR / per-feature checklist. Trim to what applies to this project. Generated by the `ux-design` plugin's `ux-foundations` skill.

## Foundations
- [ ] Uses design tokens — no hardcoded colors, font sizes, or spacing values.
- [ ] New colors meet WCAG contrast (4.5:1 body text, 3:1 large text & UI components).
- [ ] Components are built on the shared accessible primitives (Button, Field, Dialog).

## Accessibility (WCAG 2.2 AA)
- [ ] Every interactive element is reachable and operable by keyboard alone.
- [ ] Visible focus indicator on all focusable elements.
- [ ] Images have meaningful `alt` (or `alt=""` if decorative); icons-as-buttons have accessible names.
- [ ] Form fields have associated `<label>`s; errors are programmatically linked (`aria-describedby`).
- [ ] Heading levels are sequential; landmarks (`main`, `nav`, etc.) are present.
- [ ] Color is never the only signal (also use text/icon/shape).
- [ ] Touch targets ≥ 24×24 CSS px (prefer 44×44).
- [ ] Respects `prefers-reduced-motion`.

## Interaction feedback
- [ ] Every async action has loading, success, and error states.
- [ ] Lists/data views have an empty state with a clear next action.
- [ ] Status messages are announced (`aria-live` / role="status"|"alert").
- [ ] Destructive actions confirm or are undoable.

## Content / UX copy
- [ ] Error messages say what happened and how to fix it (no jargon, no blame).
- [ ] CTAs are specific verbs ("Save changes", not "Submit").
- [ ] Empty states explain value, not just absence.

## Measurement
- [ ] Core Web Vitals observed (LCP, INP, CLS) — no regressions on key pages.
- [ ] Lighthouse/axe accessibility score holds or improves.
- [ ] Key user actions emit consistent analytics events.
